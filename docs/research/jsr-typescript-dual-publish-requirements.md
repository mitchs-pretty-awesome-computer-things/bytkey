# Research: JSR TypeScript and dual-publish requirements for `@mpact/bytkey`

Date: 2026-07-10
Source: https://jsr.io/docs/publishing-packages, https://jsr.io/docs/package-configuration, https://jsr.io/docs/npm-compatibility, https://jsr.io/docs/about-slow-types, https://jsr.io/docs/using-packages, https://jsr.io/docs/with/node, https://jsr.io/docs/with/deno, https://jsr.io/docs/troubleshooting, https://jsr.io/docs/faq.

## TL;DR recommendation

- Publish TypeScript source (`.ts`) to JSR; JSR consumes source directly and transpiles it for consumers.
- Keep a separate `npm publish` path that ships compiled `.js` + `.d.ts` for Node/Bun compatibility, because JSR does **not** replace NPM for Node-based consumers.
- Use `jsr.json` for JSR metadata (`name`, `version`, `exports`, `publish.include/exclude`) and `package.json` for NPM metadata; the two files can coexist in the same repository.
- Make JSR exports point at TypeScript files (e.g. `"exports": "./src/mod.ts"`); make `package.json#exports` point at compiled files (e.g. `".": { "types": "./dist/mod.d.ts", "import": "./dist/mod.js" }`).
- Avoid CommonJS entirely; both registries should receive ESM only.
- Write explicit return types and explicit types on exported classes/variables so JSR can generate `.d.ts` for its npm compatibility layer and documentation.
- Use `node:` imports for Node built-ins; they are supported by JSR.
- Keep scope/package names lowercase, hyphenated, and within JSR length limits (`@mpact` scope is fine; `bytkey` package name is fine).

## 1. Does JSR consume `.ts` source directly, or does it expect compiled output?

JSR is designed to consume TypeScript source directly.

> "JSR supports and encourages publishing TypeScript source code rather than pairs of `.js` + `.d.ts` files." — [Publishing packages](https://jsr.io/docs/publishing-packages)

When a package is published:

- JSR stores the TypeScript source.
- For native JSR consumers (Deno, recent pnpm/Yarn), the source is type-checked and executed/transpiled by the runtime or package manager.
- For NPM-compatible consumers, JSR generates an npm-compatible tarball containing transpiled `.js` and generated `.d.ts` files on demand. This tarball is exposed through the special `@jsr` scope (e.g. `@jsr/mpact__bytkey`). — [npm compatibility](https://jsr.io/docs/npm-compatibility)

Implication for `@mpact/bytkey`:

- The JSR entrypoint should be a `.ts` file.
- The NPM entrypoint should be pre-compiled `.js` + `.d.ts` files, because `npm install @mpact/bytkey` must work out of the box on Node and Bun without requiring the consumer to transpile TypeScript.

## 2. What belongs in `jsr.json` versus `package.json`?

### `jsr.json` (or `deno.json`/`deno.jsonc`)

This is the JSR package manifest. Required fields:

- `name` — the JSR package name with scope, e.g. `"@mpact/bytkey"`.
- `version` — a valid SemVer version.
- `exports` — one string or an object mapping entrypoints to source files.

Optional fields relevant here:

- `license` — an SPDX identifier (or include a root license file).
- `publish.include` / `publish.exclude` — control which files are uploaded to JSR.

Example:

```json
{
  "name": "@mpact/bytkey",
  "version": "0.1.0",
  "license": "MIT",
  "exports": "./src/mod.ts",
  "publish": {
    "include": ["LICENSE", "README.md", "src/**/*.ts"]
  }
}
```

### `package.json`

This is the NPM package manifest and also the general project manifest for Bun/Node tooling. It should contain:

- `name` — `"@mpact/bytkey"`.
- `version` — same SemVer as `jsr.json`.
- `type: "module"` — because both JSR and the recommended NPM artifact are ESM-only.
- `exports` — pointing at compiled output for NPM consumers.
- `files` — whitelist for the NPM tarball (e.g. `dist/`, `LICENSE`, `README.md`).
- `scripts` for build/test/lint/format.
- Dev dependencies such as `typescript`, `vitest`, `oxlint`, `oxfmt`.

Example:

```json
{
  "name": "@mpact/bytkey",
  "version": "0.1.0",
  "type": "module",
  "exports": {
    ".": {
      "types": "./dist/mod.d.ts",
      "import": "./dist/mod.js"
    }
  },
  "files": ["dist", "LICENSE", "README.md"]
}
```

The two files coexist. `jsr.json` is read by `jsr publish` / `deno publish`; `package.json` is read by `npm publish` and by Bun/Node tooling.

## 3. How do JSR exports interact with Node/Bun/Deno module resolution?

### Deno

Deno has native JSR support. It resolves `jsr:@mpact/bytkey` using the `exports` field in `jsr.json` and loads the TypeScript source directly.

```ts
import { BytKey } from "jsr:@mpact/bytkey";
```

### Node and Bun (via JSR's npm compatibility layer)

Node and Bun install JSR packages through the `@jsr` scope:

```sh
npx jsr add @mpact/bytkey
# creates .npmrc: @jsr:registry=https://npm.jsr.io
# installs @jsr/mpact__bytkey
```

The install maps the package to an npm-compatible tarball generated by JSR. The tarball contains transpiled JavaScript and generated `.d.ts` files, with a `package.json` whose `exports` field is copied from `jsr.json`/`deno.json`. — [npm compatibility](https://jsr.io/docs/npm-compatibility)

> "The tarball also contains a `package.json` file that contains the `exports` field from the original `jsr.json` / `deno.json(c)` file."

Important: the JSR npm tarball's `exports` points at the transpiled files generated by JSR, not at the original `.ts` source.

### Node/Bun (via direct NPM install of `@mpact/bytkey`)

If the package is also published to NPM, `npm install @mpact/bytkey` resolves `package.json#exports`. This path is fully under our control and should point at pre-built `.js` + `.d.ts` files.

Recommended shape:

```json
{
  "exports": {
    ".": {
      "types": "./dist/mod.d.ts",
      "import": "./dist/mod.js"
    },
    "./package.json": "./package.json"
  }
}
```

Node/Bun understand this without any extra configuration.

## 4. What conventions exist for packages that publish to both JSR and NPM?

There is no single official "dual-registry" template, but the community pattern is:

1. Maintain source as TypeScript in `src/`.
2. Build to `dist/` with `tsc` (or `bun build`) for the NPM tarball.
3. Publish to JSR with `npx jsr publish` (or `deno publish`), pointing `exports` at `src/mod.ts`.
4. Publish to NPM with `npm publish`, pointing `exports` at `dist/mod.js` + `dist/mod.d.ts`.
5. Keep versions in `jsr.json` and `package.json` identical, usually by deriving both from the Git tag during CI.
6. Use `publish.include` / `files` to ensure each registry receives only what it needs:
   - JSR receives `src/**/*.ts`, `README.md`, `LICENSE`.
   - NPM receives `dist/**/*`, `README.md`, `LICENSE`.

JSR explicitly supports depending on npm packages and allows npm packages to depend on JSR packages via the `@jsr` scope, so the two registries are complementary rather than exclusive. — [Other registries](https://jsr.io/docs/other-registries)

## 5. Pitfalls around `node:` imports, package exports maps, and `@scope/name` naming

### `node:` imports

JSR supports `node:` specifiers. If a `package.json` is present, Node built-ins may also be imported without the `node:` prefix, but the explicit `node:` form is recommended because it works everywhere.

> "`node:` built-ins are supported: You can import Node.js built-ins using the `node:` scheme." — [Publishing packages](https://jsr.io/docs/publishing-packages)

For a library targeting Node/Bun/Deno, prefer web-standard APIs where possible, and use `node:` imports only when there is no standard alternative.

### Package exports maps

- JSR `exports` supports only two shapes: a single string or an object mapping entrypoint names to file paths. Conditional exports (`"import"` / `"require"` / `"types"`) are **not** used in `jsr.json`; they belong in `package.json` for the NPM artifact. — [Package configuration](https://jsr.io/docs/package-configuration)
- The JSR npm compatibility layer copies the `exports` object verbatim into the generated tarball's `package.json`. If the `exports` object contains paths that only make sense for TypeScript source (e.g. `"./src/mod.ts"`), JSR will rewrite them to the generated `.js` / `.d.ts` files in the tarball.
- Keep paths simple and avoid directory exports; JSR prefers explicit file paths.

### `@scope/name` naming

JSR scopes and names have stricter rules than NPM:

- Scope: 2–32 characters, lowercase letters, numbers, and hyphens only.
- Package name: 2–20 characters, lowercase letters, numbers, and hyphens only.
- `@mpact` (5 chars) and `bytkey` (7 chars) both satisfy these constraints. — [Publishing packages](https://jsr.io/docs/publishing-packages#creating-a-scope-and-package)

NPM scopes can be longer and allow underscores, so a name that is valid on JSR is almost always valid on NPM, but the reverse is not guaranteed.

## 6. "Slow types" and type declaration generation

JSR performs a "fast check" on TypeScript source to generate documentation and `.d.ts` files for the npm compatibility layer. It cannot run the full TypeScript compiler, so exported symbols must have explicit types or types that can be "simply inferred."

Rules that matter for `@mpact/bytkey`:

- Exported functions must have explicit return types.
- Exported classes must have explicit property types.
- Exported constants/variables should have explicit type annotations.
- No CommonJS (`export =`, `import = require(...)`).
- No module/global augmentation (`declare global`, `declare module`).
- No destructuring in exports (`export const { foo, bar } = ...`).

Example:

```ts
// Good
export class BytKey {
  private readonly secret: Uint8Array;

  constructor(secret: Uint8Array) {
    this.secret = secret;
  }

  public redeem(code: string): boolean {
    // ...
    return true;
  }
}

// Bad
export class BytKey {
  private readonly secret; // missing type
  constructor(secret: Uint8Array) {
    this.secret = secret;
  }
}
```

If slow types are unavoidable, publishing with `--allow-slow-types` is possible, but it degrades documentation, type declarations, and consumer type-checking performance. — [About slow types](https://jsr.io/docs/about-slow-types)

## 7. Publishing mechanics

- From a local machine: `npx jsr publish` opens a browser for authentication.
- From GitHub Actions: link the JSR package to the repo in JSR settings, then use a workflow with `permissions.id-token: write` and run `npx jsr publish`. No token secret is required because JSR uses OIDC. — [Publishing packages](https://jsr.io/docs/publishing-packages#publishing-from-github-actions)
- Dry run: `npx jsr publish --dry-run` verifies files, slow types, and exports without publishing.
- NPM publishing uses `npm publish` with an `NPM_TOKEN` secret; this is independent of JSR publishing.
